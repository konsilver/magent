# 容量估算与复杂度分析

为系统设计提供关键指标的数量级估算（QPS、存储量、带宽、内存）和算法/数据结构的复杂度分析，辅助架构选型和容量规划决策。

## 重要说明

- 本技能**不执行代码**，专注于数量级推算和复杂度分析。
- 估算结论须**展示完整推算过程**，不直接给出没有依据的数字。
- 估算值为数量级参考，实际值需结合真实负载测试。
- 若用户提供的前提数据不足，主动询问缺失的关键参数（DAU、读写比、数据大小等）。

## Instructions

### 容量估算场景

1. **明确输入参数**：DAU（日活用户数）、每用户每日操作次数、读写比、单条数据大小、缓存命中率等
2. **推算 QPS**：
   - `QPS = DAU × 每日操作次数 / 86400`
   - 峰值 QPS 通常取均值的 2-3 倍
3. **推算存储量**：
   - `日增存储 = 写 QPS × 单条大小 × 86400`
   - `年存储 = 日增 × 365`，给出是否需要分库分表的建议
4. **推算带宽**：
   - `带宽 = 读 QPS × 平均响应体大小`
5. **推算内存（缓存）**：
   - 按热点数据占比（通常 20% 数据承载 80% 流量）估算缓存容量
6. **给出架构建议**：基于估算结果，说明是否需要缓存层、读写分离、分片等

### 复杂度分析场景

1. **识别数据结构或算法**：一致性哈希、跳表、布隆过滤器、LSM-Tree、LRU 等
2. **给出时间/空间复杂度**：区分最好/平均/最坏情况
3. **解释在系统设计中的适用场景**：为什么这个结构适合该场景（如跳表适合有序集合的范围查询）
4. **对比替代方案**：和同类数据结构/算法的复杂度对比，说明权衡

## 常用估算基准

| 参数 | 数量级参考 |
|---|---|
| 1 天秒数 | ≈ 86,400（约 10^5） |
| 1 年秒数 | ≈ 3,000 万（3×10^7） |
| 短链接一条记录 | ≈ 500 Bytes |
| 用户头像图片 | ≈ 200 KB |
| 视频（1分钟HD） | ≈ 100 MB |
| MySQL 单表推荐上限 | ≈ 2000 万行 |
| Redis 单实例内存 | ≈ 16-64 GB |

## Inputs

- 容量估算：DAU / 操作频次 / 读写比 / 数据大小（必须提供其中若干，不足则追问）
- 复杂度分析：数据结构或算法名称 + 应用场景（必须）

## Outputs

**容量估算输出：**
- 完整推算过程（逐步展示公式和数字）
- 关键指标汇总表（QPS / 存储 / 带宽 / 内存）
- 架构建议（是否需要缓存/分片/读写分离）

**复杂度分析输出：**
- 时间复杂度（增删查，最好/平均/最坏）
- 空间复杂度
- 适用场景说明（为什么适合 / 不适合该设计场景）
- 与替代方案的复杂度对比
